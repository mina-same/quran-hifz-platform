## Key Learnings

- **Project:** quran hifz platform
- `quran-hifz-mobile` has zero API integration — all data is static mock constants in `lib/data/*.ts`, no fetch/axios anywhere. Don't assume it calls the server.
- `quran-hifz` (web) is the only app wired to `quran-hifz-server`, via relative `/api` path in `src/lib/api.ts`.
- `quran-hifz/src/quran/**` (the actual portal UI — dashboards, tables, cards) styles with a hand-rolled CSS design system in `quran.css` using `var(--green)/--surface/--border/--radius` custom properties and semantic classes (`.card`, `.stats-row`/`.stat-box`, `.tbl`/`.tbl-wrap`), NOT Tailwind — even though `src/components/ui/*.tsx` (shadcn) exists in the repo and IS Tailwind-based. New shared components for the portal pages (e.g. `quran-hifz/src/quran/components/common/Skeleton.tsx`, added 2026-07-02) must follow the `quran.css` var-based convention and reuse existing layout classes, not shadcn/Tailwind classes, or they'll look inconsistent.
- [2026-07-03] Canonical 114-surah dataset (name + ayah count) now exists at `quran-hifz-server/src/data/surahs.ts` and a byte-identical copy at `quran-hifz/src/quran/data/surahs.ts` (kept in sync manually, no shared package between the two npm projects). Sourced from quran.com API v4 `name_arabic`, cross-checked against alquran.cloud API `numberOfAyahs` — zero mismatches across all 114, total 6236 ayahs confirmed programmatically. Any future feature needing surah/ayah reference data should import `SURAHS`/`SurahMeta` from here rather than re-deriving it.
- [2026-07-03] `quran-hifz-server` Zod schemas that need `.superRefine()` for cross-field checks (e.g. XOR'd target fields) should keep the base `z.object({...})` unassigned to a `const`, build a second `const xSchemaCreate = xSchema.superRefine(...)` for create/POST, and use the original un-refined `xSchema.partial()` for update/PUT — `.superRefine()` returns a `ZodEffects` which has no `.partial()`. Matches the existing `homework.controller.ts` pattern (separate `homeworkSchema` vs `reviewSchema`), now also used in `quran-plan.controller.ts`.
- [2026-07-03] All UI-facing Arabic must be Modern Standard Arabic (Fusha), never Egyptian/colloquial (Amiya) — user explicitly checks for this. Found and fixed one dialect leak (`"من فين لفين"`, Egyptian for "from where to where", in `TeacherPlans.tsx`'s plan-range label) plus a digit-system convention violation. Two *different* number-formatting conventions coexist and must not be cross-applied: **web app** (`quran-hifz/src/lib/format.ts`'s `toAr()`/`pct()`) always renders numbers as Arabic-Indic digits (٠-٩), for both static and API-dynamic values — used pervasively (e.g. `StudentAttendance.tsx`). **Mobile app** (`quran-hifz-mobile`, no `toAr` helper exists) instead splits by data source: hardcoded/static mock arrays use Arabic-Indic string literals typed directly (e.g. `student/dashboard.tsx`'s `STATS`), while real API-backed dynamic screens (e.g. `student/attendance.tsx`) deliberately render plain Western digits with only a "٪" suffix appended — that's intentional and consistent, not a bug. When auditing/adding numeric UI text, match whichever convention the *sibling* file in that same app already uses rather than assuming one global rule.
- [2026-07-03] This web app (`quran-hifz`) has no URL-based routing for auth/portal pages — it's a single client-state app mounted at `/` (`QuranApp.tsx`): `LandingPage` → `LoginPage` → portal, switched via React state (`AuthGate`'s `step`), not TanStack Router paths. Post-login page navigation within a portal uses a `window.location.hash` read (`PortalContext.readHash`) plus `Sidebar.tsx` `.nav-item` divs (`onClick={() => showPage(id)}`). `LoginPage.tsx` has dev-account quick-fill buttons (`button.login-dev-btn`, labels "مدير"/"معلم"/"طالب"/"ولي أمر") that set the form to seeded credentials — much faster than typing email/password in E2E scripts.
- [2026-07-03] Playwright isn't installed anywhere in this repo (frontend or backend) and `npx playwright ...` alone fails ESM `import { chromium } from 'playwright'` resolution for a standalone driver script. To browser-test this app: `mkdir` a scratch dir, `npm init -y && npm install playwright@<version matching `npx playwright --version`>`, then `npx playwright install chromium`, then `node driver.mjs` from that dir.

Since `PortalContext.showPage(id)` takes only a page id (no param passing), the teacher's "ربط خطة" (connect plan) button hands off which track to prefill via a `sessionStorage` key (`PLAN_PREFILL_TRACK_KEY`, exported from `api/quran-plans.ts`) that `TeacherPlans.tsx` reads in a mount-only `useEffect` to auto-open its create modal with `targetType: "specialTrack"` pre-set — reuse this sessionStorage-handoff pattern for any future "jump to page X and pre-fill form for entity Y" need, since there's no router/query-param mechanism in this app to lean on instead.

- [2026-07-03] Added track-card grids in `quran-hifz/src/quran/pages/**/*SpecialTracks.tsx` using inline `gridTemplateColumns: "repeat(auto-fill,minmax(380px,1fr))"` with no `className="grid-collapse"` — on a phone narrower than 380px this forces a track wider than the viewport (CSS Grid doesn't shrink a fixed `minmax()` floor below the container), causing horizontal overflow/scroll instead of stacking. The user caught it by asking to "make this page more mobile friendly." Fix: any inline multi-column/auto-fill grid in this codebase must get `className="grid-collapse"` (see the `.grid-collapse` entry above) so it forces `1fr` under the 900px breakpoint — don't rely on `minmax(Xpx, 1fr)` alone to be mobile-safe just because it's `auto-fill`.
- [2026-07-03] When a form submits a nested array of objects with a backend-required field (e.g. `QuranPlan.pointRules[].label` — Zod `min(1)` server-side), always add a matching client-side pre-submit check. Skipping this on the teacher plan-builder's "add point rule" button let a user add an empty-label row, click submit, and get nothing but the generic `"بيانات غير صالحة"` message with the modal just silently staying open — cost a full extra E2E debug cycle to trace via network-response logging back to the missing `label`. Fixed by adding `if (form.pointsEnabled && form.pointRules.some((r) => !r.label.trim())) setFormError(...)` before building the request body, and trimming labels in the submitted payload.
- [2026-07-03] NEVER run `npm run seed` (or any full reseed script) without asking the user first, even when asked to "add seed data." `quran-hifz-server/src/seeds/seed.ts` starts with `Promise.all([...Model.deleteMany({})])` across every collection, and `ENV.MONGO_URI` in this project's `.env` points to a shared Atlas cluster (`mongodb+srv://.../quran-hifz`), not a disposable local DB — a full reseed would destroy any other session's/agent's current data. When asked to add demo data for a new feature, insert new documents additively (via the running server's own REST API with a real auth token, or `Model.create`) instead, and only touch `seed.ts` to add the new block for *future* full reseeds, without executing it yourself.
- [2026-07-03] `ts-node`/`ts-node-dev` in `quran-hifz-server` fails with `TS5109 (moduleResolution must be NodeNext)` when pointed at a `.ts` file **outside** `src/` (violates `tsconfig.json`'s `rootDir: "./src"`) — the error message is misleading (sounds like a global tsconfig problem) but is really just a rootDir violation. To run an ad-hoc verification script against server code (e.g. testing a new utility function), drop it inside `src/` temporarily (e.g. `src/_verify_x.ts`), run via `npx ts-node --transpile-only src/_verify_x.ts`, then delete it — don't debug the tsconfig.
- [2026-07-02] NEVER use `git stash` in this repo while other agents may be working in parallel on the same working tree (this project routinely runs backend/web/mobile agents concurrently per the "run all phases at once" decision). A `git stash -u` used to diff pre/post-change tsc errors trapped a parallel agent's in-progress `quran-hifz-server` (Phase 1 backend) and `quran-hifz-mobile` (Phase 3) changes; the subsequent `git stash pop` then aborted on unrelated `.wolf/anatomy.md`/`.wolf/memory.md` conflicts (both append-only logs that other agents update constantly), leaving ALL of my own quran-hifz edits AND the other agents' backend/mobile edits sitting only in the stash with nothing on disk. Recovered with `git checkout stash@{0} -- <path>` per file (safe — doesn't touch the working tree merge machinery) then `git stash drop`. Going forward: to check "did my change introduce new errors," run tsc on the current tree first and reason about which errors are pre-existing by checking `git diff` on just the files that error out (in my change set or not) — do NOT stash the whole tree to get a clean baseline.
- [2026-07-02] `quran-hifz/src/quran/pages/admin/AdminSpecialTracks.tsx` had `TrackCard`, `InfoRow`, and `SectionHeader` defined as nested function components inside `AdminSpecialTracks()`. Any state change on the page (typing in the search box, opening a modal, etc.) gave React a brand-new component identity each render, forcing a full unmount+remount of every card in the list instead of a prop patch — this caused the "click on student doesn't open / feels laggy / lands in the wrong place" bug (bug-081). Fix: always define list-item/card components at module scope and pass state/handlers down as props, never define a component function inside another component's render body. Check other pages for the same pattern (`TeacherSpecialTracks.tsx` also nests its `TrackCard` — lower risk there since that page has far fewer state updates, but worth hoisting if it gets a similar report).
- [2026-07-01] `quran-hifz-mobile/tsconfig.json` was silently broken — `"baseUrl": "."` without `"ignoreDeprecations": "6.0"` makes `typescript@6.0.3` abort with TS5101 before checking any file, so `tsc --noEmit` has never actually run on this project. Fixed by adding `"ignoreDeprecations": "6.0"`. This uncovered ~12 pre-existing files with real type errors unrelated to any session's work (missing `theme.cream`/`theme.greenPale` in `lib/theme.ts`, `Badge`/`StatBox` used with props their types don't declare, `number && {...}` conditional styles, a `StyleSheet.absoluteFillObject` typo, a bad `StatusBar` prop) — see bug-011 in `.wolf/buglog.json` for the exact file list. Always run `tsc --noEmit` once early in a mobile session to check this hasn't regressed, since the error is silent (exits 1 with no per-file output) and easy to mistake for "no errors."
