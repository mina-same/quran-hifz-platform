- [2026-07-10] **Mongoose gotcha, bit twice in one session:** spreading a Mongoose subdocument directly (`{ ...s }` where `s` came from `plan.schedule` / any `[SubSchema]` array field on a fetched document) does NOT give you a plain object — it copies internal Mongoose instance properties (`__parentArray`, `$__`, `_doc`, etc.) instead of the actual field values, and any `required` field then reads as `undefined` on save (triggers `ValidatorError: Path 'x' is required` even though the source document clearly had that field populated). Always call `.toObject()` on the parent document first (`plan.toObject().schedule.map(...)`), matching the existing convention already used in `quran-plan.controller.ts`'s `withTodayAssignment` — never `doc.subArray.map((s) => ({...s}))` directly off a live Mongoose doc. Hit this in both `student-plan-progress.controller.ts`'s `getStudentProgress` (silent data corruption — leaked Mongoose internals into the JSON API response) and `studentPlanReflow.ts`'s `initStudentOccurrences` (hard 500 on the very first `record` call). See [[bug-log-student-plan-progress]].
- [2026-07-10] Added `StudentPlanProgress` (`quran-hifz-server/src/models/StudentPlanProgress.model.ts`) — a per-`(plan, student)` overlay on top of the shared `QuranPlan.schedule`, tracking actual per-occurrence completion (`pending/done/partial/absent`) and a redistributable `occurrences[]` array. Deliberately NOT a full per-student plan clone — `QuranPlan` stays the single shared template; the overlay only materializes lazily on the first absence/partial-completion/manual-edit event for that student (`initStudentOccurrences` clones the plan's current schedule at that moment). Reflow (`quran-hifz-server/src/lib/studentPlanReflow.ts`'s `reflowStudentPlan`) redistributes a shortfall across that student's own remaining `pending` (non-`manualOverride`) occurrences using the same whole-page-division rule as `sliceForOccurrence`, but the pool's *last* entry must keep its ayah endpoint pinned to its `base*` fields (the plan's true `rangeEnd`, which can be mid-page) rather than the full page boundary — otherwise redistribution silently extends the plan past its real end. `manualOverride: true` (set by the per-student `updateScheduleEntry` equivalent) permanently excludes an occurrence from the reflow pool, same "pin it so automation doesn't clobber a hand-edit" pattern as the plan-level schedule. No cron/job runner exists anywhere in this backend, so reflow is purely reactive (triggered by the `record`/`reflow` endpoints), never scheduled.
